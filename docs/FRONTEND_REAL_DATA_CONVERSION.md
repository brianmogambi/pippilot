# Phase 9 — Frontend Real-Data Conversion

Converts the PipPilot UI to fully consume real backend data and makes
the three possible per-row states (`live`, `cached`, `fallback`)
visually unmistakable. No backend or schema changes — Phases 1–8
already persist every timestamp we need.

---

## Why

After Phases 1–8, the React SPA was already wired to the live
Supabase tables, but three quiet bugs remained:

1. **Silent fallbacks.** Several places replaced missing live data
   with hardcoded values without telling the user:
   - `Index.tsx` substituted the dashboard "Market Watch" widget with
     `marketSummary.slice(0, 5)` whenever the user's watchlist was
     empty, hiding the existing empty-state copy.
   - `Index.tsx` rendered `riskProfile?.max_daily_loss_pct ?? 5` as if
     it were the user's configured limit.
   - `usePipValue` exposed an `isLive` flag that only **one** call
     site (`RiskCalculator.tsx`) read. `useDailyRiskUsed` silently
     consumed static fallback pip values.
   - `useMarketData` JSDoc still claimed "Falls back to mock data"
     even though the code returned `null`.
2. **Cached vs live was invisible.** `market_data_cache.updated_at`
   exists in the schema but no hook surfaced it and no UI displayed
   it. Users had no way to tell whether a price was 30 seconds old
   or 30 minutes old.
3. **Phase 8 metadata was wired but unrendered.** `usePairAnalysis()`
   in `use-market-data.ts` duplicated `rowToAnalysis()` and silently
   dropped the explanation metadata. `SignalDetailDrawer.tsx`
   rendered the explanation under a hardcoded "AI Explanation"
   heading regardless of whether the prose was authored by Claude or
   the deterministic templates.

## Three states

| State | Color token | When | Tooltip |
|---|---|---|---|
| `live` | `bullish` (green) | Row exists in cache and `updated_at < 10 min` | "Updated N minutes ago" |
| `cached` | `warning` (yellow) | Row exists but `updated_at ≥ 10 min` (or null) | "Updated N minutes ago" |
| `fallback` | `muted` (grey) | No row exists; UI is rendering deterministic defaults | "No live data — Edge Function hasn't returned this pair yet" |

A single helper drives every decision:

```ts
// src/lib/data-freshness.ts
export const STALE_THRESHOLD_MS = 10 * 60 * 1000; // 10 minutes
export type Freshness = "live" | "cached" | "fallback";
export function freshnessOf(updatedAt, hasData, now = () => new Date()): Freshness;
```

A single visual primitive renders it:

```tsx
// src/components/ui/status-badge.tsx
<FreshnessBadge freshness={f} title="…tooltip…" />
```

`FreshnessBadge` is a one-line wrapper around the existing
`StatusBadge` with three new variants (`live` / `cached` / `fallback`)
that reuse the existing `bullish` / `warning` / `muted` colour tokens.
**No new colours, no new files for UI primitives.**

## Where each state can be seen

| Page | Widget | Freshness source |
|---|---|---|
| Dashboard (`Index.tsx`) | "Market Watch" header | Worst freshness across visible watchlist rows |
| Dashboard (`Index.tsx`) | "Market Watch" row (per-pair) | `market_data_cache.updated_at` for that pair (live rows show no badge — density) |
| Dashboard (`Index.tsx`) | "Risk Used Today" stat-card | `(default)` suffix when no `user_risk_profiles.max_daily_loss_pct` |
| Watchlist (`Watchlist.tsx`) | Page header "Market Watch" | Worst freshness across all rows |
| Watchlist (`Watchlist.tsx`) | Row (per-symbol) | `freshnessOf(row.updatedAt, !!row)`; live rows show no badge |
| Pair detail (`PairDetail.tsx`) | Page header next to pair name | Per-pair `useMarketData(pair)` |
| Pair detail (`PairDetail.tsx`) | Chart toolbar | `cached` while `useCandlesWithFetch` is auto-refreshing existing candles |
| Signal detail drawer (`SignalDetailDrawer.tsx`) | "Explanation" / "AI Explanation" heading | `pair_analyses.explanation_status` (Phase 8 metadata) |
| Risk calculator (`RiskCalculator.tsx`) | "Pip Value" cell sub-label | Existing `live` / `estimated` text — now driven by the unified `Freshness` type |

### Signal detail drawer — explanation status mapping

| `explanationStatus` | Title | Badge | Tooltip |
|---|---|---|---|
| `ai_success` | `AI Explanation` | `live` | "Generated by {model} · prompt {version}" |
| `ai_failed` | `Explanation` | `cached` | "AI call failed ({error_code}) — showing template" |
| `ai_skipped` | `Explanation` | `fallback` | "AI not configured — showing template" |
| `template_only` / null | `Explanation` | `fallback` | "Template explanation" |

## Freshness threshold

`STALE_THRESHOLD_MS = 10 * 60 * 1000` (10 minutes), defined once in
`src/lib/data-freshness.ts`. Justification: the `fetch-market-data`
Edge Function refreshes on a ~5-minute cadence (Twelve Data plan
limit), so 10 minutes is a 2× safety margin against cron skew.

To tune: change the constant. Every consumer reads it through
`freshnessOf()` so there is exactly one knob.

## Hook surface changes

| Hook | Before | After |
|---|---|---|
| `useAllMarketData()` | `Record<string, MarketData> \| null` | `MarketDataMap = Record<string, MarketData & { updatedAt: string \| null }> \| null` |
| `useMarketData(symbol)` | `MarketData \| null` | `{ data: MarketData \| null; freshness: Freshness; updatedAt: string \| null }` |
| `usePairAnalysis(symbol)` | duplicated `rowToAnalysis()` (dropped Phase 8 metadata) | calls shared `rowToAnalysis()` from `_shared/row-to-analysis.ts` |
| `usePipValue(pair)` | `{ pipValue, isLive }` | `{ pipValue, freshness }` |
| `usePipValues()` | `{ getPipValue, isLive }` | `{ getPipValue, freshness }` |
| `useDailyRiskUsed()` | `{ riskUsedPct, isLoading }` | `{ riskUsedPct, isLoading, pipFreshness }` |

The shared `MarketData` interface in `src/types/trading.ts` is
**unchanged**. The `updatedAt` field is added at the map-value level
(`MarketData & { updatedAt: string \| null }`) so the dozens of
consumers of the bare `MarketData` interface in components and utils
do not need to change.

### Manual smoke (preview server)

1. Empty `market_data_cache` → dashboard "Market Watch" shows the
   empty-state, Watchlist rows render `Demo` badges and `—` cells,
   pair-detail header shows `Demo`.
2. Insert a row with `updated_at = now()` for one symbol → that
   symbol shows `Live` (no badge on the row by design; header pill
   shows `Live`).
3. Backdate the row 30 minutes → `Cached` badge appears, tooltip
   reads "Updated 30 minutes ago".
4. Sign in without a `user_risk_profiles` row → "Risk Used Today"
   tile shows `(default)` sub-label.
5. Open a signal whose `pair_analyses.explanation_status =
   'ai_skipped'` → drawer "Explanation" header carries a `Demo`
   badge with tooltip "AI not configured — showing template".
6. Flip the row to `'ai_success'` → badge becomes `Live`, tooltip
   names the model + prompt version.
7. Flip to `'ai_failed'` with `error_code='http_429'` → badge becomes
   `Cached`, tooltip "AI call failed (http_429) — showing template".
8. Pair-detail chart with stale candles → `Cached` badge appears in
   the chart toolbar while `useCandlesWithFetch` is refetching, then
   disappears.

## Key design decisions

1. **One freshness type, one threshold.** `Freshness = "live" |
   "cached" | "fallback"` and `STALE_THRESHOLD_MS = 10 min` live in
   one tiny file. No per-page magic numbers.
2. **Reuse the existing `StatusBadge`.** Three new variants, one
   wrapper component, zero new files for UI primitives.
3. **No `MarketData` interface change.** The `updatedAt` is exposed
   as a map-value augmentation so the dozens of consumers of the
   bare `MarketData` interface do not need to change.
4. **Fix `usePairAnalysis` while we're there.** The Phase 8 metadata
   gap in the duplicated mapper was a real bug — silently dropping
   `explanationStatus` on the pair-detail page would have made the
   new badge invisible on that one route. Extracting `rowToAnalysis`
   into `_shared/` is the smallest change that fixes both call sites.
5. **The dashboard's silent watchlist substitution dies.** The
   empty-state copy already existed — Phase 9 just lets it actually
   render.
6. **No new colours.** Live = bullish green, cached = warning yellow,
   fallback = muted grey. Tooltips carry the precise meaning.
7. **No backend changes.** Phases 1–8 already persist every timestamp
   we need.
