# Phase 7 — Alert Engine v2

Adds the missing **producer** behind PipPilot's `alerts` table: a pure
rules module + a thin Edge Function runner that evaluates active signals
against the latest cached prices and inserts new alerts.

---

## Why v2

All the *consumers* of alerts already existed before Phase 7 — the
`alerts` table (`supabase/migrations/20260402081317_*.sql`), the audit
trail trigger (`20260408100100_add_alert_state_log.sql`), the lifecycle
icons in `src/pages/Alerts.tsx`, and the React Query hooks in
`src/hooks/use-alerts.ts`. What was missing was the **producer**: no
process actually inserted alerts when a signal formed, when price
crossed the entry zone, when TP1/2/3 printed, when the SL invalidated
the setup, or when the user's open risk crossed the safety threshold.
The table only ever filled if a row was hand-inserted.

Phase 7 ships that producer as:

1. A pure rules module (`src/lib/alert-engine.ts`) that classifies
   `(signal, price, prior alerts)` into `AlertCandidate[]`. Backend-safe,
   testable in vitest, no React/Supabase/DOM.
2. A verbatim Deno mirror in `supabase/functions/_shared/alert-engine.ts`
   so the runner can `import` it.
3. An Edge Function runner (`supabase/functions/evaluate-alerts/index.ts`)
   that fetches inputs, calls the engine, and persists rows behind a
   unique partial index that makes the runner idempotent.
4. An additive `useEnrichedAlerts` hook so the dashboard widget can
   surface the source signal's setup type without a separate fetch.

The `Alerts.tsx` page is **not modified** — it already renders by `type`
string with vocabulary that matches `AlertEventKind`, and the runner
writes both `event_kind` (canonical) and `type` (back-compat mirror).

## Architecture

```
                ┌──────────────────────┐
                │ src/lib/alert-engine │  pure: no React, no Supabase,
                │       (rules)        │  no DOM/Node-only globals.
                └──────────┬───────────┘
                           │ verbatim copy
                           ▼
       ┌──────────────────────────────────────┐
       │ supabase/functions/_shared/          │
       │   alert-engine.ts                    │
       │   risk-engine.ts                     │  also mirrored
       │   pip-value.ts                       │  (engine deps)
       └──────────┬───────────────────────────┘
                  │
                  ▼
       ┌──────────────────────────────────────┐
       │ supabase/functions/evaluate-alerts   │
       │   • fetch active signals             │
       │   • fetch latest market_data_cache   │
       │   • fetch user_watchlist             │
       │   • fetch prior alerts (for dedup)   │
       │   • run engine over (signal × user)  │
       │   • insert rows                      │
       └──────────┬───────────────────────────┘
                  │
                  ▼
       ┌──────────────────────────────────────┐
       │  alerts table                        │
       │  + alerts_unique_pending_event       │
       │     (user_id, signal_id, event_kind) │
       │     where status = 'pending'         │
       └──────────────────────────────────────┘
```

The engine **never** owns I/O. The runner is the only place that talks
to Supabase. This keeps the engine pure, makes it trivially mockable in
tests, and lets a future in-process consumer (e.g. the deterministic
signal engine) call it inline.

## Lifecycle event catalog

| `event_kind` | Severity | Fires when | Detector |
|---|---|---|---|
| `setup_forming` | `info` | `verdict='trade'` AND `status` ∈ `{active, monitoring}` | `detectSetupForming` |
| `entry_reached` | `info` | `\|price - entry\|` ≤ tolerance band (max of 5 pips and `0.05 × ATR`) | `detectEntryReached` |
| `confirmation_reached` | `info` | Status flip from `monitoring/active` → `ready/triggered/confirmed` | `detectConfirmationReached` |
| `tp1_reached` | `info` | long: `price ≥ tp1`; short: `price ≤ tp1` | `detectTpReached(_, _, 1)` |
| `tp2_reached` | `info` | long: `price ≥ tp2`; short: `price ≤ tp2` | `detectTpReached(_, _, 2)` |
| `tp3_reached` | `info` | long: `price ≥ tp3`; short: `price ≤ tp3` | `detectTpReached(_, _, 3)` |
| `invalidation` | `warning` | long: `price ≤ SL`; short: `price ≥ SL`; OR `invalidation_reason` set | `detectInvalidation` |
| `risk_breach` | `critical` | `openRiskPct > 5%` (safety) OR `realizedLossPct > maxDailyLossPct` (profile) OR `openRiskPct > maxDailyLossPct` | `detectRiskBreach` (composes `calculateOpenRiskUSD` + `RISK_THRESHOLDS` from Phase 6) |

The full mapping lives in `SEVERITY_BY_KIND` (`src/lib/alert-engine.ts`)
and is asserted by the test suite.

Title and message are generated by pure formatters in the engine, so
re-running with identical inputs produces byte-identical text.
Format: `"{Setup} · {DIRECTION} {event} on {pair}{maybe at price}"`.

## Dedup contract

Two layers, both must hold:

1. **Engine layer (`prior.fired` filter).** The runner pre-fetches every
   `(user_id, signal_id, event_kind)` already in the alerts table, builds
   a `Map<"${user_id}:${signal_id}", Set<AlertEventKind>>`, and passes it
   to `evaluateSignalAlerts` as `prior.fired`. The engine drops any
   candidate whose kind is in the set.
2. **DB layer (unique partial index).**
   `alerts_unique_pending_event` on `(user_id, signal_id, event_kind)
   WHERE status = 'pending'` (added in `20260412100000`). Race-safe under
   concurrent runners — two processes inserting the same `(user, signal,
   kind)` will produce one row, the other gets a `duplicate key` error
   that the runner swallows.

The engine also writes `dedupe_key = '${signal_id}:${event_kind}'` on
every row, indexed by `alerts_dedupe_key_idx`, as a stable hash for
client-side analytics.

## Source-tagging

Each alert row is tagged with:

- `signal_id` (FK to `signals.id`) — already existed pre-Phase 7. Always
  set, even for `risk_breach` (the runner anchors each user's breach
  alert to the most-recent active signal so the FK is satisfied — see
  *Open questions* in the plan).
- `analysis_run_id` (FK to `generation_runs.id`) — added in
  `20260412100000`. Best-effort: the runner picks the latest successful
  `function_name = 'generate-signals'` run when persisting. A future
  phase will add a `pair_analyses.run_id` FK so this can be looked up
  per-signal instead of globally.
- `type` — back-compat mirror of `event_kind`. The Alerts page reads
  `type` and is therefore unchanged.

## Scheduling

Three options, in order of preference:

### 1. pg_cron (recommended once enabled)

```sql
-- Requires pg_cron and pg_net extensions (Supabase enables on demand).
select cron.schedule(
  'evaluate-alerts',
  '*/5 * * * *',
  $$
  select net.http_post(
    url := 'https://<project-ref>.functions.supabase.co/evaluate-alerts',
    headers := '{"Authorization":"Bearer <service-role-key>","Content-Type":"application/json"}'::jsonb,
    body := '{"source":"pg_cron"}'::jsonb
  );
  $$
);
```

Migration not yet shipped — operator action.

### 2. Chained call (this is what ships today)

`fetch-market-data/index.ts` does a fire-and-forget `fetch()` to
`evaluate-alerts` after a successful market-data refresh. Failure of
the chained call never poisons the parent response.

### 3. Manual

```bash
supabase functions invoke evaluate-alerts
# or from the dashboard "Refresh" button
```

## Frontend contract

Existing hooks in `src/hooks/use-alerts.ts` are **untouched**:

- `useAlerts()` — full list
- `useDashboardAlerts(limit)` — recent N
- `useUnreadAlertCount()`
- `useMarkAlertRead()`, `useMarkAllAlertsRead()`

New additive hooks:

```ts
export type EnrichedAlert = Alert & {
  event_kind: AlertEventKind | null;
  analysis_run_id: string | null;
  signal: {
    setup_type: string | null;
    direction: string | null;
    confidence: number | null;
    entry_price: number | string | null;
    verdict: string | null;
  } | null;
};

useEnrichedAlerts(opts?: { kinds?: AlertEventKind[] })
useDashboardAlertsEnriched(limit = 5)
```

Both use a PostgREST FK join (`signals!alerts_signal_id_fkey(...)`) so
the dashboard widget can render setup type / direction next to each
alert without a second round-trip. No type regeneration needed —
`event_kind` and `analysis_run_id` are read via narrow casts (same
pattern as Phases 3 and 5).

## TODOs for future phases

  once `pg_cron` and `pg_net` are enabled on the project.
- **Sync script for the Deno mirror** — `src/lib/{alert,risk}-engine.ts`
  and `supabase/functions/_shared/{alert,risk}-engine.ts` are kept in
  sync by hand. Same trade-off Phase 6 acknowledged for `risk-engine`.
  A small `scripts/sync-shared.ts` could codegen the `_shared/` copies.

  per-signal instead of using the global "latest successful run"
  heuristic.
- **Per-user alert preferences** — mute kinds, snooze pair, channel
  selection (in-app / email / push). The schema can grow without engine
  changes.
- **Realized-loss tracking** — `RiskContext.realizedLossUSD` is wired
  but the runner currently passes `0`. A future phase will sum closed
  trade P&L for the day from `trade_journal_entries`.
- **`alert_state_log` UI surface** — render the audit trail in the
  Alerts page (the trigger already populates the table).
- **Watchlist-free fan-out** — today only watchlist users get alerts.
  A future phase could add an opt-in "alert me on every active signal"
  per-user toggle.
- **Confirmation diffing** — `evaluateSignalAlerts` accepts a
  `priorSignal` parameter for status-flip detection, but the runner
  currently passes `null`. Wiring the prior signal state requires a
  small `signals_history` table or a `previous_status` column.
